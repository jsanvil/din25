    1.5.1 ✍️ Proyecto: Angular Events. Parte 1. Listado de eventos

    Vamos a ir implementando poco a poco un proyecto de gestión de eventos (deportivos, conciertos, etc.). Según vayamos viendo en clase los diferentes contenidos de Angular, se irá implementando una actividad asociada para reforzar el aprendizaje.

    Tomando como referencia del ejemplo realizado en 1.5 Introducción a Angular, sigue los pasos para crear una aplicación similar con los siguientes requisitos:

    1. Acepta la invitación a la actividad en GitHub Classroom: https://classroom.github.com/a/_-bmS6mP

      • Selecciona tu usuario del listado.
      • Acepta la actividad.

    2. Clona el repositorio en tu máquina local

      Puedes utilizar VS Code para clonar el repositorio o la terminal con el comando:

      Clonar el repositorio
      git clone <url-repositorio>

    3. Crea un proyecto llamado angular-eventos.

    4. Configura el proyecto para que funcione con Electron.

      Guía completa en: 1.4 Trabajar con Angular y Electron

    5. Crea un componente llamado eventos-show. Contendrá casi todo el HTML que necesitaremos en el ejercicio.

    6. Crea una interfaz llamada Evento (evento.ts) dentro del directorio interfaces/.

      Representará los datos de un evento (título, imagen, fecha, descripción y precio).

    9. No olvides instalar Bootstrap y Angular Material

    10. Listando los eventos con @for

      En el componente EventosShow, crea un array de objetos Evento (el tipo será Evento[]). Crea al menos 2 eventos con todos sus datos. La imagen la puedes dejar como cadena vacía por ahora, más adelante veremos como mostrarla. La fecha debe ser una cadena en formato ‘YYYY-MM-DD’.

    11. Si el array de eventos está vacío, con las directivas @if y @else, muestra el mensaje "No hay eventos disponibles" dentro de un <h3>, por ejemplo.

    12. Puedes realizar todos los commit que necesites, en el último pon el mensaje "v1.0.0".

      Si por algún motivo necesitas realizar cambios, una vez termines sube el número de versión patch

      Por ejemplo: 1.0.0 ___ 1.0.1 ___ 1.0.2 ___ etc.

      Referencia: Semantic Versioning 2.0.0

      Given a version number MAJOR.MINOR.PATCH, increment the:

      • MAJOR -> incompatible breaking changes
      • MINOR -> new feature/functionality
      • PATCH -> bug fixes

      Por lo tanto para corregir fallos (bug fixes) se debe incrementar el último número.

      TAG

      Lo habitual es etiquetar con TAGs para establecer las distintas versiones. Pero en este caso GitHub Classroom no permite el uso de etiquetas.

    13. Comprueba que los cambios se han subido correctamente al repositorio remoto.
